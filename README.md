# My Code Generator

Агент для генерации и отладки Python-кода с использованием локальной LLM и Docker-контейнера для безопасного исполнения кода.
Проект реализует паттерн ReACT (Reasoning + Acting) с поддержкой trace-augmented self-debugging —
метода автоматической отладки, при котором после неудачного выполнения кода генерируется трассировка
переменных через AST-инструментацию и передаётся модели для анализа.

## Режимы работы

| Режим | Описание | Точка входа |
|-------|----------|-------------|
| **CLI** | Интерактивный диалог с агентом в терминале | `src/main.py` |
| **Web UI** | Веб-интерфейс с SSE-стримингом через FastAPI | `src/app.py` |
| **HumanEval** | Бенчмарк генерации кода (164 задачи) | `run_humaneval.py` |
| **QuixBugs** | Бенчмарк исправления ошибок (40 задач, 4 seed, 2 версии) | `run_quixbugs.py` |

## Требования

- **Python** 3.11+
- **Docker** (для песочницы выполнения кода)
- **llama-server** или совместимый OpenAI API сервер (llama.cpp, vLLM и т.д.)
- **GPU** с достаточным объёмом VRAM для выбранной модели (рекомендуется 24 ГБ для Qwen3-14B-Q4)

## Установка

### 1. Клонирование и создание виртуального окружения

```bash
git clone <repository-url>
cd my_code_generator

python -m venv .venv

# Linux / macOS
source .venv/bin/activate

# Windows
.venv\Scripts\activate
```

### 2. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 3. Сборка Docker-образа песочницы

```bash
docker build -f Dockerfile.sandbox -t code-sandbox:latest .
```

Образ основан на `python:3.11-slim` и включает библиотеки для вычислений
(numpy, scipy, pandas, scikit-learn, matplotlib, networkx и др.),
но не содержит сетевых или системных пакетов.

```bash
cp .env.example .env
```

Отредактируйте `.env` под вашу конфигурацию (см. [Конфигурация](#конфигурация)).

## Запуск

### 1. Запуск LLM-сервера

Перед использованием агента необходимо запустить LLM-сервер.

**llama-server (llama.cpp):**

```bash
llama-server \
  -m /path/to/Qwen3-14B-Q4_K_M-GGUF.gguf \
  --port 8080 \
  -c 32768 \
  --parallel 2 \
  -ngl 99
```

Ключевые параметры:
- `-c` — размер контекстного окна (общий)
- `--parallel` — число параллельных слотов (размер слота = `-c` / `--parallel`)
- `-ngl` — число слоёв, вынесенных на GPU (`99` = все)

Убедитесь, что `LLAMA_BASE_URL` в `.env` указывает на запущенный сервер.

### 2. Интерактивный CLI

Диалоговый режим в терминале. Агент принимает задачи на естественном языке,
генерирует код, выполняет его в Docker-песочнице, отлаживает при необходимости
и сохраняет результат в файл.

```bash
# Работа в текущей директории
python src/main.py

# Указать рабочую директорию проекта
python src/main.py /path/to/your/project
```

**Команды в CLI:**
- `quit` — выход
- `clear` — сброс истории разговора

### 3. Веб-интерфейс

FastAPI-сервер с веб-интерфейсом и SSE-стримингом вызовов инструментов.

```bash
python src/app.py
```

Интерфейс доступен по адресу: **http://127.0.0.1:7860**

**API-эндпоинты:**

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/` | Веб-интерфейс чата |
| `GET` | `/info` | Информация о модели и параметрах |
| `GET` | `/browse` | Выбор рабочей директории (диалог ОС) |
| `POST` | `/chat` | Отправка сообщения агенту (SSE-стрим) |

**Параметры POST `/chat`:**

```json
{
  "message": "Напиши функцию сортировки",
  "working_dir": "/path/to/project",
  "agent_history": [],
  "max_steps": 8,
  "max_tokens": 16384,
  "temperature": 0.2
}
```

### 4. Бенчмарк HumanEval

Бенчмарк генерации кода на 164 задачах из набора HumanEval.
Поддерживает автоматическое возобновление после прерывания.

```bash
# Быстрая проверка: 3 задачи (первая, средняя, последняя)
python run_humaneval.py --mode debug

# Полный прогон: все 164 задачи (возобновляемый)
python run_humaneval.py --mode full

# С trace-augmented self-debugging (аблация B1+trace)
python run_humaneval.py --mode full --trace-debug
```

Альтернативный запуск через пакет:

```bash
python -m run_humaneval --mode debug
python -m run_humaneval --mode full --trace-debug
```

**Аргументы:**

| Аргумент | Значения | Описание |
|----------|----------|----------|
| `--mode` | `debug`, `full` | `debug` — 3 задачи для проверки; `full` — все 164 |
| `--trace-debug` | флаг | Включить трассировку выполнения при ошибках |

Результаты сохраняются в директорию `humaneval_results/`.

### 5. Бенчмарк QuixBugs

Бенчмарк исправления ошибок на ~40 задачах из набора QuixBugs.
Сравнивает две версии агента:
- **B0** — plain self-debug (без трассировки)
- **B1** — trace-augmented self-debug (с трассировкой выполнения)

Каждая задача выполняется с 4 фиксированными seed-значениями (7, 42, 137, 2718)
для статистической достоверности.

```bash
# Быстрая проверка: 2 задачи × 4 seed × 2 версии = 16 вызовов
python run_quixbugs.py --mode debug

# Полный прогон: ~40 задач × 4 seed × 2 версии ≈ 320 вызовов (возобновляемый)
python run_quixbugs.py --mode full

# Запуск для одного seed (возобновляемый, интегрируется с full)
python run_quixbugs.py --mode seed --seed 7
python run_quixbugs.py --mode seed --seed 42
python run_quixbugs.py --mode seed --seed 137
python run_quixbugs.py --mode seed --seed 2718
```

Альтернативный запуск через пакет:

```bash
python -m run_quixbugs --mode debug
python -m run_quixbugs --mode full
python -m run_quixbugs --mode seed --seed 42
```

**Аргументы:**

| Аргумент | Значения | Описание |
|----------|----------|----------|
| `--mode` | `debug`, `full`, `seed` | `debug` — 2 задачи для проверки; `full` — все задачи × все seed; `seed` — все задачи для одного seed |
| `--seed` | `7`, `42`, `137`, `2718` | Seed для режима `seed` (обязательный) |

Результаты сохраняются в директорию `quixbugs_results/`.

**Визуализация результатов:**

После завершения бенчмарка можно построить график кривых сходимости —
зависимость доли решённых задач от номера итерации отладки для агентов B0 и B1.
Скрипт принимает summary JSON-файл, который генерируется бенчмарком автоматически.

```bash
# График для одного seed
python plot_convergence.py quixbugs_results/quixbugs_seed7_summary.json

# График с агрегацией по всем seed (среднее ± std)
python plot_convergence.py quixbugs_results/quixbugs_full_summary.json
```

Результат сохраняется рядом с исходным файлом в формате PNG
(например, `quixbugs_seed7_summary_convergence.png`).

## Архитектура

### ReACT-агент

Ядро системы — цикл ReACT (Reasoning + Acting), реализованный в `src/agent.py`.
На каждом шаге модель обязательно вызывает один из инструментов (`tool_choice="required"`),
а затем получает результат для следующего шага.

**Доступные инструменты:**

| Инструмент | Описание |
|------------|----------|
| `execute_code` | Выполнение Python-кода в Docker-песочнице (таймаут 10 сек) |
| `read_file` | Чтение файла из рабочей директории |
| `write_file` | Запись файла (только после успешного `execute_code`) |
| `list_files` | Просмотр содержимого директории |
| `update_session_memory` | Сохранение прогресса в память сессии |
| `respond_to_user` | Финальный ответ пользователю (завершает цикл) |

**Порядок работы при генерации кода:**

```
execute_code → write_file → update_session_memory → respond_to_user
```

Файлы с кодом (`*.py`, `*.js`, `*.ts` и др.) невозможно записать, пока `execute_code`
не вернёт успешный результат — это гарантирует, что сохраняется только проверенный код.

### Trace-Augmented Self-Debugging

При неудачном выполнении кода в `execute_code` активируется механизм трассировки:

1. Код разбирается в AST
2. В стратегические точки (присваивания, вызовы функций, генераторы) внедряются вызовы `__trace_log__`
3. Инструментированный код выполняется повторно
4. Трассировка сжимается (первые и последние строки, середина суммируется)
5. Результат трассировки добавляется к сообщению об ошибке для анализа моделью

### Управление контекстным окном

Модуль `src/context_manager.py` обеспечивает работу в пределах контекстного окна локальной модели:

1. **Обрезка длинных результатов** — сохраняется начало и конец, середина заменяется на `[N chars omitted]`
2. **Скользящее окно** — при превышении бюджета удаляются самые старые сообщения (системный промпт никогда не удаляется)

### Память

Двухуровневая система памяти:

- **PROJECT_MEMORY** (`memory/` в проекте) — описание проекта, стека, конвенций; добавляется к системному промпту
- **SESSION_MEMORY** (`~/.coding_agent/memory/<project>/`) — записи прогресса сессий; автоочистка по возрасту

## Docker-песочница

Код выполняется в изолированном Docker-контейнере с многоуровневой защитой:

### Уровень 1 — AST-валидация (`src/sandbox/validator.py`)

Перед запуском код анализируется на уровне AST:
- **Блокируются** опасные модули: `subprocess`, `os`, `pickle`, `ctypes`, `shutil`
- **Определяются** несовместимые со средой модули: `socket`, `requests`, `flask`, `django`, `sqlalchemy`

### Уровень 2 — Docker-изоляция

```
--network=none            # нет сети
--memory=64m              # лимит памяти
--cpus=0.5                # лимит CPU
--read-only               # файловая система только для чтения
--tmpfs /tmp:noexec,nosuid  # /tmp без exec
--user=nobody             # непривилегированный пользователь
--cap-drop=ALL            # все capabilities отключены
--security-opt=no-new-privileges
--pids-limit=32           # защита от fork-бомб
timeout 10s               # ограничение по времени
```
